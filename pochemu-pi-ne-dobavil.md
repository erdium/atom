# Почему Pi сознательно отказался от некоторых функций

Основано на блог-посте Марио Цехнера (автор Pi) и pi.dev.

---

## 1. MCP (Model Context Protocol)

**Что это:** Стандартный протокол от Anthropic для подключения внешних инструментов к LLM-агентам. MCP-сервер регистрирует список тулов, агент их вызывает через JSON-RPC.

**Почему не включили:**

- MCP-серверы дампят описания ВСЕХ своих инструментов в контекст при подключении. Playwright MCP — 21 тул, 13.7k токенов. Chrome DevTools MCP — 26 тулов, 18k токенов. Это 7–9% контекстного окна уходит до начала работы.
- Вместо этого: обычный CLI-скрипт с README. Агент читает README только когда понадобится — платишь токены только за нужное. Прогрессивное раскрытие.
- CLI — уже стандарт. `curl + bash + README` работают везде, без протокола и лишних зависимостей.
- Если очень хочется MCP — есть утилита `mcporter`, которая оборачивает MCP-сервер в CLI-команду.

## 2. Sub-agents (саб-агенты)

**Что это:** Возможность для агента запускать внутри себя другой экземпляр агента для выполнения подзадачи.

**Почему не включили:**

- Саб-агент — чёрный ящик. Нет observability в то, что он делает. Контекст передаётся плохо — оркестратор решает, что отдать саб-агенту, и у вас нет контроля.
- Если саб-агент ошибается, дебажить больно — не видно полной переписки.
- Вместо этого: Pi может запустить себя через bash (в tmux или отдельной сессии). Полная наблюдаемость, можно даже зайти в ту же tmux-сессию и поработать вместе с агентом.
- Главная проблема саб-агентов — их используют для сбора контекста. Это неправильный подход. Собирай контекст в отдельной сессии, создавай артефакт, потом работай с ним в чистой сессии. Артефакт полезен и для следующей фичи, и видно всё.
- Спавнить несколько саб-агентов для параллельной реализации фич — антипаттерн. Кодовая база превратится в свалку.

## 3. Permission popups (подтверждения на каждое действие)

**Что это:** Окна с вопросом «разрешить агенту прочитать файл / выполнить команду / отправить запрос?»

**Почему не включили:**

- YOLO by default. Pi работает с полным доступом: файловая система, любые команды, без проверок.
- Если агент умеет писать код И запускать код — game over, любой popup — security theater. Единственная реальная защита — запускать агента в контейнере.
- Все равно все работают в YOLO-режиме, чтобы получать результат. Так зачем делать вид, что popup'ы хоть что-то защищают?
- Единственный способ реально защититься — отрезать весь сетевой доступ у окружения, где работает агент. Но тогда агент бесполезен.
- Simon Willison (dual LLM pattern) признаёт: любое решение «очень плохое» и добавляет огромную сложность. Если LLM может читать данные, выполнять код и ходить в сеть — это whack-a-mole с уязвимостями.

## 4. Plan mode (режим планирования)

**Что это:** Встроенная возможность попросить агента сначала подумать и составить план, не трогая файлы.

**Почему не включили:**

- Избыточен. Можно просто сказать агенту обдумать проблему вместе с вами, не запуская bash. Это и так работает.
- Если нужен план между сессиями — пиши в файл PLAN.md. Агент прочитает, обновит, будет сверяться. План версионируется вместе с кодом.
- В Claude Code Plan Mode в итоге всё равно пишет план в markdown-файл на диск. Разница с Pi — только в observability. В Pi видно сразу, какие файлы агент реально посмотрел, а какие пропустил. В Claude Code оркестратор спавнит саб-агента, и вы не видите, что он делает.
- Если нужно защититься от изменений во время планирования: `pi --tools read,grep,find,ls` — режим только для чтения.

---

## 5. Встроенные to-do (списки задач)

**Что это:** Автоматически отслеживаемый агентом список завершённых и оставшихся задач.

**Почему не включили:**

- Опыт показывает: to-do листы больше путают модель, чем помогают. Добавляют состояние, которое модель должна отслеживать и обновлять — больше возможностей для ошибок.
- Если нужно отслеживать задачи — пиши TODO.md. Агент прочитает, обновит, отметит чекбоксы. Просто, видимо, под вашим контролем.

---

## 6. Background bash (фоновые команды)

**Что это:** Агент запускает процесс (dev-сервер, тесты) и продолжает работу, не дожидаясь его завершения.

**Почему не включили:**

- Фоновые процессы добавляют сложность: трекинг процессов, буферизация вывода, очистка при завершении, отправка ввода.
- В Claude Code это ломалось при compaction — агент забывал о всех фоновых процессах и не мог их найти.
- Вместо этого: **tmux**. Запустил dev-сервер в tmux-сессии — видишь весь вывод, можешь зайти и поковыряться, можешь даже дебажить вместе с агентом.
- Tmux даёт всё, что нужно: аргумент CLI для листинга активных сессий, полная observability, прямое взаимодействие.

---

## Общая философия: Primitives, not features

Pi — минимальный agent harness:

- **Системный промпт:** меньше 1000 токенов (против 10k+ у Claude Code).
- **Набор инструментов:** read, write, edit, bash. И ещё grep/find/ls для read-only режима.
- **Всё остальное** — через расширения, skills, пакеты, README и внешние инструменты.

Фичи — это примитивы, а не встроенный функционал. Всё, что не нужно в 100% сессий, должно быть расширением, чтобы не раздувать контекст и не усложнять ядро.
